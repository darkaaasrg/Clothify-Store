# Опис архітектури Kubernetes-додатку "Clothify Store"

Ми моделюємо розгортання веб-застосунку **Clothify Store** (Flask/Python), який призначений для онлайн-шопінгу та управління відгуками покупців. Усередині кластера сервіс розміщено в ізольованому просторі імен (Namespace) під назвою `clothify-prod`.

Зовнішній трафік надходить на домен `shop.clothify.ua`, проходить через **Cloud Load Balancer** до **Ingress Controller**, який на основі правил маршрутизації (Path: `/`) перенаправляє запити на внутрішній **Service** типу `ClusterIP`. Цей сервіс виконує роль внутрішнього балансувальника та розподіляє навантаження між **3-ма Pod-ами**, які я заклала для забезпечення відмовостійкості (згідно з принципами Practice 5) та високої доступності.

У цій схемі процес доставки коду автоматизований:
1. Код зберігається в **git**-репозиторії на GitHub (`github.com//clothify-store`).
2. Після кожного push спрацьовує **CI/CD** pipeline (GitHub Actions), який збирає Docker-образ.
3. Образ пушиться в **container registry** (Docker Hub) під тегом `clothify-app:1.0.0`.
4. Під час старту або оновлення Deployment-а, Kubernetes завантажує цей образ із registry прямо в **Pod-и** для виконання.
