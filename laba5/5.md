# Практика №5: Надійність та відмовостійкість API

В межах даної практики було реалізовано комплекс механізмів для захисту серверної частини від перевантажень та забезпечення цілісності даних на стороні клієнта.

## 1. Серверні механізми (Backend)

* **Кореляція запитів (Tracing)**: Додано Middleware, який призначає кожному вхідному запиту унікальний заголовок `X-Request-Id`. Це дозволяє ефективно логувати та відстежувати повний шлях запиту від клієнта до сервера.
* **Rate Limiting**: Впроваджено обмеження частоти запитів для захисту від спаму та DoS-атак. Встановлено ліміт: **15 запитів на 10 секунд** для однієї IP-адреси.
* **Обробка перевантажень**: При перевищенні ліміту сервер повертає статус `429 Too Many Requests` разом із заголовком `Retry-After: 5`, що вказує клієнту час очікування перед наступною спробою.
* **Моніторинг**: Створено ендпоінт `GET /api/health`, який повертає `{ "status": "ok" }` для швидкої перевірки працездатності системи.



## 2. Клієнтська стійкість (Frontend Resilience)

Для забезпечення стабільної роботи інтерфейсу при нестабільному з'єднанні реалізовано наступні механізми у функції `fetchWithResilience`:

* **Retry Policy**: Система автоматично виконує до **3 повторних спроб** у разі невдалого запиту.
* **Exponential Backoff & Jitter**: Паузи між повторами розраховуються експоненційно (`2^спроба * 500мс`), а додавання випадкової затримки (**джитеру**) запобігає синхронному навантаженню на сервер від багатьох клієнтів одночасно.
* **Timeouts**: Використано `AbortController` для встановлення жорсткого ліміту очікування відповіді — **2 секунди**. Це запобігає «зависанню» інтерфейсу при відсутності відповіді від сервера.



## 3. Цілісність даних та обмежений режим

* **Ідемпотентність POST-запитів**: Для операції додавання товару до кошика використовується заголовок `Idempotency-Key`. Сервер зберігає результати запитів у `idem_store`, що гарантує: навіть при повторному надсиланні одного і того ж запиту (чекаючи на статус 201), товар буде додано до бази лише **один раз**.
* **Degraded Mode (Обмежений режим)**: У разі реєстрації 3 послідовних збоїв (наприклад, помилки 429 або таймаути), фронтенд автоматично переходить у захисний режим:
    * Кнопка дії дизейблиться (стає сірою).
    * Активується таймер зворотного відліку на **10 секунд**.
    * Виводиться інформаційний банер про перевантаження системи.

---

**Результат**: Реалізована архітектура дозволяє системі Clothify Store залишатися стабільною при високому навантаженні, забезпечує коректність фінансових операцій та надає користувачу зрозумілий зворотний зв'язок у разі технічних проблем.
